# Webベース・プレゼンテーションスライドツール

[DIST.53「スライドデザイン道場」レポート](https://creatorconnect.jp/blog/DIST53-CreatorConnect) におけるセッション **「『スライドは動かすな』は本当か？ 発表資料の理解を助けるモーションデザインの手法5選」**（池田泰延氏／株式会社ICS）で紹介された、**Web技術ベースの自作スライドツール**をインスピレーション源としたプロジェクトです。理解を助けるモーションと、ブラウザならではの表現・運用を両立することを目指します。

### 運用・プロダクト上の前提

- **「ビルド」とは何を指さないか／指すか**  
  - **利用者の原稿（`.md` / `.html`）については事前ビルドを不要とする。** リポジトリやサーバに置いたファイルを、**ブラウザが実行時に読み取り・解釈**してスライドとして表示する。Marp CLI や静的サイトジェネレータでデッキを事前に HTML へ焼くことを**必須にしない**。  
  - **本ライブラリ（プレゼン viewer／ランタイム）自体の開発**については、Bundler・TypeScript コンパイル・npm の `build` など **一般的なビルド工程を前提としてよい**（配布物がブラウザで動けばよい）。

- **Marp 形式を第一級で扱う**  
  [Marp](https://marp.app/) の**基本的なスライドの切り方・メタ情報の扱い**に則る。具体的には YAML フロントマター、`---` によるページ区切り、ページ単位のディレクティブ（`<!-- -->` 形式を含む）など、Marp として広く共有されているルールを**スライド構造の正**とする。  
  原稿は **そのまま編集・配置でき、手軽にプレビュー・本番投影できる**ことは本ツールの**魅力の中核**とする。VS Code の Marp 拡張など既存エコシステムと原稿を共有しやすい。

- **HTML 原稿のスライド単位（わかりやすいルール）**  
  Marp 以外に、素の HTML でもデッキを書けるようにする。その場合の対応関係は次で固定する。  
  1. **スライドデッキのルート**：単一の親要素（例: `<div data-slide-deck>` または `<article data-slide-deck>`）。属性名は実装時に確定する。  
  2. **1 枚のスライド**：ルート要素の**直下**にある **`<section>` 要素 1 つ**がスライド 1 ページに相当する。  
  3. **入れ子**：スライドとして数えるのは**直下の `<section>` のみ**とする（`<section>` 内にさらに `<section>` を置く場合の扱いは、原則「1 スライド内のマークアップ」として解釈し、ページ分割には使わない）。  
  4. **ルート外**：デッキルートの外側の HTML は、プレゼン UI に含めないか、仕様で別途定める（共通ヘッダ等が必要ならルート内先頭スライドやオーバーレイで扱う想定）。

## 1. 背景・コンセプト（記事との対応）

記事では、モーションの問題の本質が「目立たせるための過剰演出」にあり、**理解の補助**として使えば情報伝達に役立つ、とされています。紹介された5つのテクニックは、本ツールの**表現要件・ユースケース**として取り込みます。

| 記事での手法 | 本ツールでの想定 |
|-------------|-----------------|
| 段階表示で現在地を示す | 箇条書き・図形の段階的表示（リビール） |
| スポットライトで注目点を示す | スクリーンショット等の一部強調（マスク／減光） |
| スライド間の接続型アニメーション | 前後スライドで共通要素をつなぐ遷移 |
| コード差分を動きで示す | 変更前後の対比表示（アニメーションまたはレイアウト） |
| 動画を埋め込む | スライド内への動画埋め込み・再生制御 |

セッション後半で言及された**自作Webスライドツール**の特徴は、次のとおり要件化します。

## 2. 機能要件

### 2.1 コア（必須に近い）

- **ブラウザで完結**  
  - ローカルまたは配信されたURLを開くだけで起動・操作できること。  
  - プレゼンモード（全画面またはブラウザ全画面）を想定。

- **スライドソース（いずれも原稿の事前ビルド不要）**  
  - **Markdown**：Marp 形式に準拠した区切り・フロントマター・ディレクティブでスライド構造を表現し、ブラウザ実行時にレンダリングする（上記「運用・プロダクト上の前提」参照）。  
  - **HTML**：上記の **`<section>` = 1 スライド** ルールに従ったファイルを、実行時に読み込みそのまま（または最小限のラップのみで）表示する。

- **テーマ：ライト／ダークの切り替え**  
  - 会場の明るさ・投影環境に合わせて切り替え可能であること。

- **Webタイポグラフィ**  
  - **プロポーショナルメトリクス**、**禁則処理**、**和欧間スペース** など、CSS／フォント機能で実現できる範囲の日本語組版を重視する（可読性・投影時の読みやすさ）。

- **リアルタイムカメラ**  
  - `getUserMedia` 等により、**ブラウザからカメラ映像をライブで取得・表示**できること。  
  - 用途例：プレゼンター顔出し、実物デモ、会場向けPiP（画中画）など。  
  - 利用者がデバイス・解像度を選べること（可能な範囲で）。

### 2.2 理解を助けるモーション（段階的実装）

- キーボード／リモート操作で **段階表示**（1クリックごとに要素を追加表示）。  
- **スポットライト**相当のUI（領域指定またはプリセット）。  
- **スライド遷移**で文脈がつながるアニメーション（共通要素の維持を意識した設計）。  
- **コードブロック**向け：差分・ハイライト・段階表示のいずれか以上。  
- **動画埋め込み**：ファイルまたはURL、再生・ループ・ミュート等の制御。

### 2.3 AI・編集体験（記事記載の延伸）

- **自然言語での修正指示**  
  - 例：「このスライドのタイトルを短く」「コードブロックを2段階で表示」など。  
  - 実装形態は後続設計とする（ローカルLLM、外部API、ルールベースの補助など）。  
  - 最低限の要件：**編集対象（Markdownまたはメタデータ）への反映パス**を設計に含める。

## 3. 非機能要件

- **セキュアコンテキスト**  
  - カメラ・マイクは **HTTPS** または **localhost** でのみ安定して利用可能なため、開発・本番のURL設計に明記する。

- **パフォーマンス**  
  - スライド枚数が増えても操作・遷移が実用的な応答性を保つこと。  
  - カメラストリームは解像度・フレームレートを抑え、メインコンテンツと競合しないこと。

- **アクセシビリティ**  
  - キーボード操作、フォーカス、コントラスト（記事でも言及のある投影時の視認性）を考慮。

- **オフライン優先（任意だが推奨）**  
  - 会場ネットワークに依存しすぎない構成（静的アセット中心など）を検討。

- **原稿の事前ビルド非依存との両立**  
  - 初回読み込み時のパースコストは許容し、必要なら Web Worker やキャッシュで体感を改善する。**md / html 原稿を事前コンパイルしなくてよい**方針は維持する。

---

## 4. 技術的前提（初期案・確定ではない）

- フロントエンド：モダンブラウザ向け（Vanilla／フレームワークは実装フェーズで選定）。  
- **Marp**：ブラウザ上では `@marp-team/marp-core` 等、Marp 互換のランタイムで Markdown → スライド用 DOM を生成する想定（**原稿 md の事前ビルドは不要**。ライブラリ本体の bundling は可）。  
- **HTML デッキ**：`fetch` 等で取得したドキュメントから `data-slide-deck` ルートを特定し、直下の `<section>` をページ列として扱う。
- カメラ：`MediaDevices.getUserMedia`、表示は `<video>` または WebGL／Canvas（必要に応じて）。

---

## 5. スコープ外（初版では扱わない可能性が高いもの）

- **デッキ原稿の事前ビルドを利用者に強いる運用**（Marp CLI 等で md を必ず HTML に変換してからでないと表示できないこと）。※開発者がライブラリを publish するためのビルドは対象外。  
- PowerPoint／Keynote ネイティブファイルの直接読み込み。  
- サーバーサイドでの同時編集（必要になれば別イテレーション）。

---

## 6. 成功基準（MVPの目安）

1. ブラウザで起動し、**配置した Marp 形式 Markdown を原稿の事前ビルドなしで**複数スライドとして順に表示できる。  
2. （HTML デッキ）**ルート直下の `<section>` 列**がページ順に表示される。  
3. ライト／ダークを切り替えられる。  
4. 日本語本文で禁則・和欧間などの指定が効いている（ブラウザ・フォントの制約内）。  
5. カメラを許可するとリアルタイム映像がスライド画面内（またはオーバーレイ）に表示される。  
6. 少なくとも1種類の「段階表示」または「遷移アニメーション」が動作する。

## 実装済み機能

### コア機能

| 機能 | 説明 |
|------|------|
| Marp Markdown 読み込み | `.md` ファイルを事前ビルドなしでブラウザ上でスライドに変換 |
| HTML デッキ読み込み | `<section>` 要素ベースの HTML ファイルをスライドとして表示 |
| ライト/ダークテーマ | ワンキーで切替可能（`T` キー） |
| カスタムテーマプリセット | 6種類のテーマを切替可能 |
| 日本語タイポグラフィ | プロポーショナルメトリクス（palt）、禁則処理（kinsoku）、和欧間スペース（waei） |
| カメラオーバーレイ | getUserMedia によるリアルタイムカメラ表示。形状（円/矩形）、サイズ（S/M/L）、ドラッグ移動対応 |
| フラグメント/段階表示 | クリック/キーごとに要素を段階的に表示するリビールアニメーション |
| View Transitions | 連続スライドで共通画像がある場合に自動的に View Transitions API で滑らかに遷移 |
| プログレスバー | 画面上部にスライド進捗を表示（`P` キーで表示/非表示） |
| スライド一覧（Grid） | 全スライドのサムネイル一覧をオーバーレイ表示、クリックでジャンプ |
| ハッシュベースページナビ | URLの `#ページ番号` でスライド位置を保持・共有 |

### プレゼンテーション支援

| 機能 | 説明 |
|------|------|
| プレゼンタービュー | 別ウィンドウで発表者ビューを表示。BroadcastChannel でスライド同期 |
| タイマー | 経過時間（elapsed）/時計（clock）/非表示の3モード。5つの表示位置（ツールバー, 左上, 右上, 左下, 右下） |
| ブラックアウト/ホワイトアウト | 画面を黒（`B`）または白（`W`）で覆う。任意のキーで解除 |
| スポットライト | マウス追従のスポットライト効果。スクロールで半径調整（`S` キー） |
| ペンツール | フリーハンド描画、Shift+ドラッグで直線、スライドごとに描画保存、取り消し（Ctrl+Z）、5色、3段階の太さ |
| ズーム/拡大鏡 | ドラッグで範囲選択して拡大表示、背景にバックドロップ、クリックで戻る（`Z` キー） |
| コード構文ハイライト | Shiki による構文ハイライト（遅延読み込み） |

### 操作

| 機能 | 説明 |
|------|------|
| キーボードショートカット | 豊富なショートカットキー（`?` でヘルプ画面表示） |
| ツールバー切替 | 右クリック / 2本指タップでツールバー表示/非表示 |
| フルスクリーン | `F` キーまたはボタンで全画面切替 |

### 配布・運用

| 機能 | 説明 |
|------|------|
| スタートページ | ファイルドロップ、URL入力、サンプルデッキ（ダウンロード/プロンプトボタン付き） |
| PDF/印刷エクスポート | Grid一覧画面から Print / PDF 出力 |
| 共有URL | クリップボードにスライドURLをコピー（現在のページ番号付き） |
| Service Worker | オフラインサポート（sw.js 登録） |
| GitHub Actions デプロイ | CI/CDによる自動デプロイ対応 |
| レスポンシブデザイン | 各種画面サイズに対応 |
| 4種類のトランジション | slide（スライド）、fade（フェード）、zoom（ズーム）、none（なし） |

| キー | 動作 |

## 開発

### 必要環境

- Node.js
- npm

### セットアップ

```bash
npm install
```

### 開発サーバー起動

```bash
npm run dev
```

### プロダクションビルド

```bash
npm run build
```

### ビルドプレビュー

```bash
npm run preview
```

### 型チェック

```bash
npx tsc --noEmit
```
